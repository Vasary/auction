# Документация Frontend (React + TypeScript) — Клиент Аукционов

Данный документ описывает архитектуру, механизмы работы и логику клиентской части системы аукционов.

## 1. Технологический стек

- **Framework**: React 18+
- **Язык**: TypeScript (строгая типизация всех событий и моделей)
- **Сборщик**: Vite
- **Стилизация**: Tailwind CSS + Shadcn UI (компоненты в `ui/src/components`)
- **Маршрутизация**: React Router v6

## 2. Архитектура и структура кода

Код фронтенда расположен в директории `ui/src`.

### Основные директории:
- `components/`: Общие UI-компоненты (Button, Card, Input и др.).
- `lib/utils.ts`: Вспомогательные функции (форматирование валют, объединение классов Tailwind).
- `pages/`: Страницы приложения.
    - `auctions/AuctionList.tsx`: Список доступных тендеров.
    - `auctions/AuctionDetail.tsx`: Основной экран аукциона с WebSocket-взаимодействием.
    - `auctions/AuctionLogin.tsx`: Вход в аукцион (выбор ID компании и пользователя).

---

## 3. Работа с WebSocket на клиенте

Основная работа с аукционом происходит в режиме реального времени через WebSocket.

### Жизненный цикл соединения:
1. **Подключение**: При открытии страницы аукциона (`/auctions/:tenderId`) устанавливается соединение по адресу `ws://<host>/ws/:tenderId`.
2. **Инициализация**: Сразу после `onopen` сервер присылает событие `snapshot`. Клиент использует его для первичной отрисовки состояния (текущая цена, шаг, статус).
3. **Обработка событий**:
    - `snapshot`: Полное обновление состояния.
    - `price_updated`: Обновление текущей цены и лидера.
    - `finished`: Перевод интерфейса в режим "Аукцион завершен" (блокировка ввода ставок).
4. **Реконнект**: В случае обрыва соединения реализован механизм автоматического переподключения (через 2 секунды) для обеспечения отказоустойчивости.

## 4. Бизнес-правила и логика ставок

Логика работы аукциона едина для всей системы. Подробное описание правил (UUID, минорные единицы, правила валидации) доступно в общем документе:

👉 **[Бизнес-правила и проверки](business-rules.md)**

### Особенности на стороне клиента:
- **Валидация**: Перед отправкой сообщения `place_bid` фронтенд выполняет первичную проверку (цена ниже текущей, кратность шагу) для улучшения UX.
- **Конкурентность**: Поскольку цена может измениться в любой момент, клиент должен быть готов к тому, что сервер отклонит ставку, даже если локальная валидация прошла успешно.
- **Обработка ошибок**: Ошибки от сервера (например, `rate_limit_exceeded`) выводятся пользователю напрямую.

---

## 5. Обработка ошибок

Система обрабатывает следующие типы ошибок:
1. **Ошибки подключения**: Индикация статуса WS (Connecting / Connected / Disconnected).
2. **Ошибки бизнес-логики (от сервера)**:
    - `bid_not_lower`: Ставка выше текущей.
    - `bid_not_aligned`: Не соблюден шаг аукциона.
    - `rate_limit_exceeded`: Слишком частые ставки от одного участника.
3. **Ошибки доступа**: Если компания не зарегистрирована как участник тендера, сервер проигнорирует или отклонит ставку.

---

## 6. Интерфейс участника

- **Панель управления**: Отображает текущую цену и ID лидера.
- **Лог событий**: Выводит историю всех входящих сообщений от сервера для отладки и прозрачности процесса.
- **История ставок**: Список всех подтвержденных ставок по данному тендеру (запрашивается через HTTP API при загрузке и обновляется при событиях).
